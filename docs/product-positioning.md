# 产品定位：AI 个人研究工作台

溯知 TraceMind 的核心定位是：**AI Personal Research Workbench，AI 个人研究工作台**。

它服务的人不是只想要一句答案的人，而是那些正在长期理解一个问题、一个领域、一个技术路线或一个研究方向的人。你可能是研究生、独立研究者、工程团队里的技术负责人、产品研究员、投资分析师、开源作者，或者任何需要把大量资料变成判断的人。

## 不是「再问一次 AI」

通用聊天模型擅长即时问答，但个人研究不是一问一答就结束。研究会跨越多天、多周甚至多年：

- 今天读到的论文，三周后可能成为某个判断的证据。
- 今天被否定的路线，半年后可能因为新模型或新数据重新变重要。
- 今天看似无关的图表，可能是理解一个技术分歧的关键。

所以溯知不把聊天当终点，而把聊天放回研究现场：每次追问都应该能连接主题、论文、节点、证据和已有记忆。

## 不是「更漂亮的文献列表」

文献管理器解决的是收藏、标注和引用。它非常重要，但研究者还需要回答更高一层的问题：

- 这些论文到底在解决同一个问题，还是只是关键词相似？
- 哪些论文构成了主线，哪些只是分支？
- 一个方法的核心假设是什么？
- 证据支持到什么程度？哪里还只是推断？
- 如果我要写一篇综述、技术判断或研究备忘录，结构应该如何展开？

溯知把论文组织成研究节点，而不是让它们停留在列表里。

## 不是「信息流」

研究者不缺更新提醒，缺的是安静而可控的工作台。溯知不追求让你不停刷新，而是帮助你在一个主题里逐渐建立：

- 研究地图
- 论文候选池
- 证据板
- 节点文章
- 阶段判断
- 会话记忆
- 可导出的研究产物

## 核心对象模型

| 对象 | 含义 |
| --- | --- |
| Topic | 一个长期研究主题，例如「端到端自动驾驶」或「具身智能任务规划」 |
| Paper | 论文及其元数据、PDF、图表、公式和引用 |
| Evidence | 可回溯的证据片段，包括文字、图片、表格、公式、引用关系 |
| Node | 主题里的研究节点，通常对应问题、方法、机制、局限或趋势 |
| Judgment | 基于证据形成的阶段性判断，允许保留不确定性 |
| Memory | 主题长期记忆，包括用户偏好、研究连续性和过往决策 |
| Conversation | 围绕当前主题和证据展开的追问，不是孤立聊天 |
| Artifact | 可导出的节点文章、研究简报、报告和参考材料 |

## 适合谁

- 正在做毕业论文、综述、开题或长期方向积累的学生和研究者。
- 想跟踪一个技术领域演化的工程师和技术负责人。
- 需要把论文证据转成产品、策略或投资判断的分析者。
- 想自托管研究数据和模型配置的开源用户。

## 不适合谁

- 只需要快速搜索一个事实。
- 不关心证据来源，只想要流畅文本。
- 希望 AI 自动替自己完成最终学术判断。
- 需要完整商业知识库权限系统和团队级审计套件。

溯知最重视的是个人研究者的掌控感：资料可以被追踪，判断可以被修正，主题可以长期生长。

## 产品北极星

溯知的北极星不是「生成更多内容」，而是：

> 让个人研究者在任何一个重要判断上，都能回到证据、上下文和当时的推理路径。

因此，溯知判断一个功能是否值得做，会优先看它是否增强了以下能力：

- 让证据更容易被找到。
- 让不确定性更容易被看见。
- 让主题结构更容易被延续。
- 让用户更容易修正 AI 和自己的判断。
- 让下一次研究不是从零开始。

## GitHub 用户应该如何理解这个项目

如果你从开源项目的角度看溯知，可以把它理解成几层能力的组合：

- 一个 React + Vite 的研究工作台界面。
- 一个 Express + Prisma 的研究 API 和数据层。
- 一个面向论文发现、PDF 抽取、证据归档、节点建模的后端服务集合。
- 一个通过 Omni Gateway 管理模型供应商、角色模型和提示词的模型治理层。
- 一套面向个人研究者的产品理念：证据优先、记忆优先、人类判断优先。

这也是为什么 README 不只展示启动命令，还要解释定位、边界、横向对比和研究初心。一个好的研究工具不应该只让开发者知道怎么跑起来，也应该让用户知道它为什么值得被信任。
