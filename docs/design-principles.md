# 设计原则

溯知的设计目标不是让界面看起来复杂，而是让研究过程更可控、更可追溯、更容易沉淀。

## 1. Evidence First

每一个重要判断都应该尽量连接到证据。证据可以是论文段落、图表、公式、表格、引用关系，也可以是研究者手动加入的备注。

这意味着溯知不会把「流畅回答」当作最高目标。更好的输出应该满足：

- 能指出依据来自哪里。
- 能标记不确定性。
- 能区分事实、解释、推断和建议。
- 能被后续证据修正。

## 2. Memory Over Chat

聊天是交互方式，不是研究结构。溯知把对话放入更大的主题记忆里：

- 用户在关心什么问题。
- 当前主题有哪些研究阶段。
- 哪些节点已经形成判断。
- 哪些证据被多次引用。
- 哪些偏好和约束应该在下次生成时继续生效。

这样，下一次追问不再是从空白开始。

## 3. Nodes, Not Folders

文件夹适合存放资料，但不适合表达研究问题。溯知使用节点组织研究：

- 一个节点可以代表一个问题。
- 一个节点可以代表一个方法族。
- 一个节点可以代表一个争议或局限。
- 一个节点可以连接多篇论文和多种证据。

节点让研究从「我收藏了什么」转向「我理解了什么」。

## 4. Workbench, Not Feed

信息流会制造焦虑，工作台应该制造掌控感。溯知的核心界面应该让用户看见：

- 当前主题的地图。
- 当前阅读焦点。
- 右侧上下文助手。
- 可引用证据。
- 已形成和待验证的判断。

用户进入溯知时，应该感觉自己回到了一个未完成但有秩序的研究现场。

## 5. Model Neutrality

研究者不应该被单一模型绑定。溯知通过 Omni Gateway 管理不同模型角色：

- 语言模型用于阅读、总结、对话和写作。
- 多模态模型用于图表、公式和视觉证据。
- 角色路由用于节点作者、视觉读者、主题架构师等任务。

模型可以换，研究结构应该保留下来。

## 6. Human Judgment Stays Central

溯知不会宣称 AI 自动完成研究。它更像研究助理、资料管理员和结构化伙伴：

- AI 可以推荐候选论文。
- AI 可以整理证据。
- AI 可以生成初稿和问题列表。
- AI 可以提示矛盾和不确定性。
- 最终判断仍由人完成。

这是溯知与许多「一键生成答案」工具最重要的区别。

## 7. Multilingual By Default

研究者的资料和表达经常跨语言。溯知支持八种界面语言，并保留中英双语和多语言扩展的设计空间。语言切换不应该只是按钮，而应该服务于阅读、理解和写作。

## 8. Documentation Is Part Of The Product

对开源研究工具来说，文档不是附属品。README、快速开始、横向对比、架构说明和路线图共同决定用户是否愿意相信这个项目。

溯知的文档应该保持四个习惯：

- 首屏说清楚它解决什么问题。
- 快速开始必须真实可执行。
- 架构说明要暴露边界，而不是制造神秘感。
- 路线图要区分已经稳定、正在推进和明确不做的内容。

这也是溯知参考成熟 GitHub 项目文档结构的原因：优秀 README 不是堆满口号，而是让用户快速形成判断。
