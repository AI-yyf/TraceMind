溯知要解决的是一个很真实的研究困境：一次研究进展、一次论文发布、一次模型更新，通常都不足以让你看清一个研究方向。

今天的 AI 研究生态很热闹，也很嘈杂。热点传播得很快，摘要生成得很快，跟进趋势也很容易，但这并不等于真正理解什么在解决问题。溯知围绕一个更难也更重要的问题设计：能不能让 AI 去追踪文献、积累证据、沉淀判断，并在此基础上成为你最忠诚、最严谨的研究助手？

它不是把每次提问都当成一次全新的聊天，而是把一个主题变成可持续生长的研究现场。你可以在这里发现论文、筛选主线、抽取 PDF 证据、建立研究节点、形成阶段判断，并让后续追问始终回到已经积累好的材料之上。

## 项目介绍

溯知是一个 AI 个人研究工作台。它适合研究生、独立研究者、工程师、技术负责人、分析师，以及所有需要把大量论文和资料整理成判断的人。

如果你正面对这些问题，溯知会更有价值：

| 你正在经历 | 溯知提供什么 |
| --- | --- |
| 论文越来越多，但主线越来越模糊 | 主题地图、节点图谱、关键论文集合、真实研究阶段 |
| AI 回答很流畅，但你找不到依据 | 绑定论文、PDF、图表、公式、引用与证据片段的回答 |
| 好问题散落在聊天记录和临时笔记里 | 围绕主题沉淀的研究工作台、长期记忆与可导出产物 |
| 研究越来越像追热点，积累却很弱 | 从真实材料中长出来的长期主题，而不是一次性摘要 |

## 初心

研究的失败，很多时候不是因为没有信息，而是因为理解没有积累起来。

一个常见的研究过程往往是这样的：
- 今天读到一篇看起来很关键的论文
- 三天后发现了一篇相反的结果
- 两周后记得结论，却忘了支撑结论的图和实验
- 一个月后，整个问题散落在标签页、收藏夹和半成品笔记里

通用聊天工具很擅长“回答”，却不擅长保存这些更重要的东西：
- 这个判断为什么成立
- 它到底由哪些证据支持
- 哪些地方仍然不确定
- 这个方向是怎样一步步演化成现在的

溯知坚持四件事：
- `证据优先`：强判断必须能回到材料本身
- `记忆优先`：工作台应该记住主题、节点和已形成的判断
- `结构优先`：论文不能只变成更长的收藏夹，而要变成研究节点
- `人类判断优先`：AI 是助手，不是替代者

## 亮点

- `主题页展示真实研究结果`：阶段、节点图谱、关键论文和研究进展来自真实积累，而不是先造一个“研究规划阶段”。
- `节点页是研究视图，不是单篇论文页`：一个节点会把核心问题、关键论文、证据链、方法、发现、限制、争议和研究判断组织成可快速扫清方向的入口。
- `证据始终可见`：论文、PDF、图表、公式、引用和抽取片段都尽量靠近最后的结论。
- `追问始终有上下文`：后续问题发生在主题和节点里，而不是在空白聊天框里重新开始。
- `默认可自托管`：模型配置、密钥和研究数据可以掌握在你自己的环境里。
- `为长期积累而设计`：它服务的是几周、几个月，甚至更长的研究弧线。

## 快速启动

前置要求：
- Node.js `18+`
- npm `9+`
- Python `3.10+`
- 至少一个本地可用的模型提供方密钥

启动后端：

```bash
cd skills-backend
npm install
cp .env.example .env
npm run db:generate
npm run dev
```

启动前端：

```bash
cd frontend
npm install
npm run dev
```

默认地址：
- 前端：`http://localhost:5173`
- 后端健康检查：`http://localhost:3303/health`

Docker 方式：

```bash
docker compose up --build
```

## 教程：前 15 分钟

1. 启动前后端，打开应用。
2. 在设置页配置至少一个模型提供方。
3. 创建一个你真的想长期追踪的主题，而不是随手写的演示关键词。
4. 运行论文发现，不要急着全盘接受结果。
5. 把真正属于主题主线的论文纳入，把“看起来相关”的论文排除出去。
6. 打开一个节点研究视图，先看结构化研究页，再决定是否深入原始论文。
7. 提一个真正能检验节点的问题，比如：`这个分支里最薄弱的证据是什么？`
8. 把结果导出，或者继续把新论文和新判断沉淀回主题里。

## 流程说明

溯知希望把研究过程沉淀成一个循环，而不是一次性生成：

1. `发现论文`：从学术来源中拉起候选池。
2. `筛选与准入`：决定哪些论文进入当前主题，哪些暂时不进入。
3. `抽取证据`：把 PDF 中的文本、图表、公式、表格和引用变成可复用对象。
4. `建立节点`：围绕问题、方法、机制、局限、争议和转折组织主题结构。
5. `形成判断`：写出当前证据支持什么、哪里薄弱、下一步该质疑什么。
6. `继续追问`：让 AI 基于已有主题上下文回答，而不是从零即兴发挥。
7. `导出产物`：把工作整理成节点文章、研究简报、综述素材或报告草稿。
8. `回到主题记忆`：让下一次研究不是重新开始。

## 对比

溯知并不想替代所有研究工具，它更像是把它们之间缺失的那一层补起来。

| 工具 | 最擅长 | 溯知所在的位置 |
| --- | --- | --- |
| Zotero | 收集、标注、引用论文 | 把文献进一步组织成节点、证据链和持续演化的判断 |
| NotebookLM | 围绕给定材料问答 | 把问答放进长期主题、研究进度和节点结构中 |
| Elicit | 检索、筛选、综述流程 | 更关注个人持续研究，而不是一次性的综述任务 |
| Perplexity | 快速带来源的答案 | 把一次性答案沉淀成主题记忆和后续研究结构 |
| Obsidian / Notion | 个人笔记与组织 | 增加论文追踪、证据绑定和研究视图 |
| ChatGPT / Claude | 推理、写作、对话 | 给模型一个有证据、有记忆的研究工作台 |

## 开源基础与参考

溯知不是重新发明每一层，而是站在成熟开源生态之上继续搭建：
- `React` 与 `Vite` 负责前端工作台
- `Express` 与 `Prisma` 负责研究 API 和数据层
- `SQLite`、`PostgreSQL`、`Redis` 提供存储与部署基础
- `PyMuPDF` 与本地脚本负责 PDF 抽取
- `OpenAI`、`Anthropic`、`Google` 等模型通过后端运行时接入
- `arXiv`、`OpenAlex`、`Crossref`、`Semantic Scholar` 等提供开放学术来源

在文档组织和开源表达上，这一版 README 也参考了 `Supabase`、`Dify`、`LangChain`、`Immich`、`Next.js`、`Visual Studio Code`、`Excalidraw` 和 `Open WebUI` 等项目的优点。我们学习的是它们如何清楚地回答四个问题：这是什么、为什么有必要、怎么跑起来、边界在哪里。

## 适合谁

溯知适合你，如果你：
- 需要持续跟踪一个研究方向，而不是只看一次结果
- 需要比较论文之间的关系，而不是只做收藏
- 需要写综述、技术判断、研究简报或报告
- 希望自托管研究数据和模型配置
- 愿意把最终判断权牢牢留在自己手里

如果你只需要这些东西，溯知可能并不适合：
- 一次性的事实查询
- 不关心证据链、只想要流畅答案
- 通用企业知识库
- 让 AI 替你做最终专业判断的系统

## 结束

一次研究进展，往往很难让人真正看清一个研究方向。更困难的是，当整个 AI 研究生态都在追逐热点、跟随趋势和表面新颖性时，真正解决问题的方法反而更容易被噪声遮住。

溯知想做的事情很朴素：让 AI 去追踪文献、积累证据、支持追问，并在这个过程中成为你最忠诚、最严谨的助手。不是让它比研究更响亮，而是让它帮助你更清楚地看见研究本身。
