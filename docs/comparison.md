# 横向对比

本页用于帮助用户理解溯知的位置。它不是为了贬低其他工具，而是说明：每类工具解决的问题不同，溯知补的是「AI 个人研究工作台」这一层。

参考对象基于各产品公开说明、官方入口和公开 GitHub/文档页面，访问日期为 2026-04-27。

## 先给结论

如果你只想收藏、标注和引用论文，优先使用 Zotero。如果你只想围绕一组上传资料问答，NotebookLM 很直接。如果你正在做系统综述里的检索和字段抽取，Elicit 更垂直。如果你需要快速获得带来源答案，Perplexity 很高效。如果你需要自由写作、编程和推理，ChatGPT / Claude 更通用。

溯知更适合另一种场景：你要长期理解一个研究方向，并希望论文、证据、问题、节点、模型输出和个人判断都能留在同一个可追溯的工作台里。

## 总览

| 工具 | 更像什么 | 最擅长 | 典型缺口 | 溯知的定位 |
| --- | --- | --- | --- | --- |
| [Zotero](https://www.zotero.org/) | 文献管理器 | 收集、管理、标注、引用文献 | 不负责把论文组织成研究节点和长期判断 | 与 Zotero 互补，承接文献之上的研究结构 |
| [NotebookLM](https://notebooklm.google/) | 源材料问答笔记本 | 围绕上传资料进行问答、摘要和生成 | 更偏资料内问答，主题生命周期和论文发现不是核心 | 将源材料问答放进主题、节点、证据和记忆系统 |
| [Elicit](https://elicit.com/) | AI 系统综述助手 | 检索论文、抽取字段、支持综述流程 | 更偏任务型综述表格和筛选流程 | 更偏个人长期研究现场和节点化理解 |
| [Perplexity](https://www.perplexity.ai/) | 联网答案引擎 | 快速搜索、综合网页来源、给出答案 | 答案通常是一次性的，不天然形成个人研究记忆 | 把答案沉淀为主题结构、证据链和可回看判断 |
| Obsidian / Notion | 笔记和知识库 | 自由记录、页面组织、个人知识管理 | 论文证据、模型治理和研究任务流通常需要自己搭建 | 把研究对象、证据抽取和 AI 追问做成一体化工作台 |
| ChatGPT / Claude | 通用 AI 助手 | 推理、写作、代码、解释和对话 | 研究对象、证据链和长期主题结构需要用户自己维护 | 将模型能力嵌入研究工作台，而不是让对话孤立存在 |

## 与 Zotero：不是替代，而是上层工作台

Zotero 的优势是稳定、成熟、开放，适合文献收集、标注、引用和参考文献管理。溯知不应该试图重做这一层。

溯知更适合回答 Zotero 之上的问题：

- 这些文献如何构成一个研究问题？
- 哪几篇是主线，哪几篇是旁支？
- 一组论文共同支持了什么判断？
- 哪些证据可以写进节点文章或研究简报？

理想使用方式是：Zotero 管文献资产，溯知管研究结构。

## 与 NotebookLM：从资料问答走向研究现场

NotebookLM 让用户围绕给定来源提问，是很适合做资料内理解的工具。溯知吸收这种「源材料优先」的理念，但把重点放得更长：

- 主题不是一次上传，而是长期演进。
- 论文不只是来源，而是候选、准入、证据和节点关系的一部分。
- 对话不只是问答，而是能回写主题记忆。
- 研究产物不只是摘要，而是节点文章、研究判断和导出包。

## 与 Elicit：从综述任务走向个人研究记忆

Elicit 很适合系统综述里的检索、筛选、抽取和表格化比较。溯知也重视论文发现和证据抽取，但目标更像个人研究台：

- 支持多主题长期积累。
- 支持节点化阅读和研究地图。
- 支持工作台对话、Prompt Studio 和模型配置。
- 支持将证据、判断、记忆和生成产物放在一起。

如果你的目标是完成一次结构化文献综述，Elicit 可能更直接；如果你的目标是长期理解一个研究方向，溯知更贴近工作台。

## 与 Perplexity：从答案引擎到可继承判断

Perplexity 擅长快速联网搜索并合成答案。它适合「我现在想知道什么」。溯知更关心「我长期如何理解这个问题」。

差异在于：

- Perplexity 的核心是答案；溯知的核心是研究对象。
- Perplexity 的上下文通常围绕一次查询；溯知围绕一个长期主题。
- Perplexity 强在网页综合；溯知强在论文、PDF、证据、节点和个人记忆。

## 与 Obsidian / Notion：从自由笔记到研究对象系统

Obsidian 和 Notion 的优势是灵活。它们适合写笔记、建页面、做个人知识库，也很适合承载研究者自己的写作习惯。溯知不是要取代这种自由空间，而是补上研究工作流里更结构化的一层：

- Topic、Paper、Evidence、Node、Judgment 是一组明确的研究对象。
- PDF、图表、公式和引用不是普通附件，而是证据链的一部分。
- 模型不是孤立聊天窗口，而是通过 Omni Gateway 和 Prompt Studio 进入任务上下文。
- 导出产物可以回到 Obsidian、Notion、Markdown 或正式报告中继续写作。

更理想的关系是：溯知负责研究证据和结构，笔记工具负责个人表达和长期写作。

## 与通用聊天模型：把模型放回结构里

ChatGPT、Claude 等模型非常强，但研究者需要自己维护上下文、来源和版本。溯知的价值不是比模型更聪明，而是给模型一个更好的工作环境：

- 哪些材料可以用。
- 哪些证据被信任。
- 当前问题处于哪个研究阶段。
- 输出应该遵守什么结构。
- 哪些地方必须标记不确定性。

通用模型像一位能力很强的临时合作者；溯知让它进入一个有档案、有证据、有记忆的个人研究办公室。

## 溯知自己的边界

溯知仍然不是万能工具：

- 它不替代专家审稿。
- 它不保证模型输出永远正确。
- 它不应该吞掉你的所有私人资料后变成黑盒。
- 它不追求把所有知识管理功能都塞进一个产品。

它最应该做好的事，是把个人研究中的资料、证据、判断和追问放在一个可追溯的工作台里。

## 选择建议

| 你的主要任务 | 更适合的起点 |
| --- | --- |
| 管理参考文献、生成 citation、同步 PDF 标注 | Zotero |
| 围绕几份上传资料快速问答 | NotebookLM |
| 做一次系统综述的检索、筛选和字段抽取 | Elicit |
| 快速查一个开放网页问题 | Perplexity |
| 写作、编程、解释概念、开放式对话 | ChatGPT / Claude |
| 组织长期研究方向、证据链、节点和阶段判断 | 溯知 |

溯知最好的使用方式不是孤立存在，而是和这些成熟工具协作：Zotero 管文献资产，NotebookLM/通用模型补充临时理解，Obsidian/Notion 承载自由写作，溯知负责把研究问题、证据和判断组织成可以延续的工作台。
